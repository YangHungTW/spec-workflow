## 1. Approach

### 1.1 Scope

Teach `bin/scaff-seed`'s `plan_copy` enumerator to land `.claude/settings.json` in the consumer repo for `init` and `migrate` modes, with a read-merge-write atomic pattern that preserves any pre-existing consumer-owned keys and writes a `.bak` before mutating. `update` mode stays unchanged (parity with team-memory: user-mutable file). Source: `.claude/settings.json` in the source repo; destination: `.claude/settings.json` in the consumer repo. Merge precedent is `bin/scaff-install-hook` (cited from `.claude/rules/common/no-force-on-user-paths.md`).

### 1.2 Why one task

The chore PRD checklist has 4 items, but they form a single coherent edit: one enumerator change in `plan_copy`, one merge implementation in the dispatcher arm, and one regression test (`t114_*.sh`) that exercises both paths (fresh-install + merge). Splitting would force the test to land in a separate commit from the implementation it covers — a known anti-pattern. Folded into T1.

## 3. Risks

- **JSON-merge edge cases** — the consumer file may be malformed JSON (hand-edited). Mitigation: behave like `bin/scaff-install-hook` — parse with stdlib `json`, on `JSONDecodeError` either treat as `{}` and back up the malformed file, or refuse loudly with a `.bak` written first. Pick the same posture as the existing precedent; do not invent a third behaviour.
- **`update` mode regression** — accidentally extending the new copy to `update` mode would clobber a consumer's edits to settings.json. Mitigation: t114 must include an `update`-mode assertion that `update` does NOT copy `.claude/settings.json` (parity with team-memory).
