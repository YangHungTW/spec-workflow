# Request

**Raw ask**: Fix `scaff-seed init` so the Stop hook lands in `<consumer>/.claude/settings.json` (the file Claude Code actually reads) and stops orphaning a `settings.json` + `.bak` at the consumer project root.

**Context**: Reproduced 2026-04-26 on a fresh empty consumer at `/Users/yanghungtw/Projects/meridian` immediately after `scaff-seed init`. Step 7's Python merge correctly writes `<consumer>/.claude/settings.json` (SessionStart only). Step 10 then shells out to `bin/scaff-install-hook add SessionStart …` and `add Stop …` from inside `consumer_root`; `scaff-install-hook` hardcodes `p = "settings.json"` (lines 66 and 109) — a bare filename with no `.claude/` prefix — so it operates on `<consumer>/settings.json` at the project root instead. Net effect: two divergent settings files post-init, the Stop hook never reaches Claude Code, and a stray `settings.json` plus `settings.json.bak` snapshot are left at the consumer root. User's stated preference is fix Option B: change the helper's default target to `.claude/settings.json` so the bare-filename literal goes away and the helper aligns with the file Claude Code actually reads.

**Success looks like**: After `scaff-seed init` on a fresh empty consumer, only `<consumer>/.claude/settings.json` exists (no root-level `settings.json` or `settings.json.bak`), and that file contains both the `SessionStart` and `Stop` hook entries pointing at `.claude/hooks/session-start.sh` and `.claude/hooks/stop.sh` respectively. Re-running `scaff-seed init` is idempotent.

**Out of scope**: Reworking the Step 7 Python merge logic; redesigning hook-entry shape or command prefixes; introducing a new `--settings-path` CLI surface beyond what's needed to make the default correct; cleaning up already-orphaned `settings.json` / `.bak` files on existing consumer checkouts (a one-time manual `rm` is acceptable).

**UI involved?**: no
