## Analysis notes

### Check 4 — Injection attacks (primary axis)

`cmd` path:
- `cmd` is validated against a hardcoded allow-list (`cmd_is_in_allow_list`) in
  `dispatch()` at line ~462 before any script building occurs; the allow-list
  contains only known safe command slugs (e.g. "implement", "prd", "plan").
- `cmd` is then passed through `shell_single_quote_escape` (line 298 / 509),
  which replaces every `'` with `'\''`, preventing escape from the surrounding
  single-quote boundary in the generated shell script body and clipboard string.
- The T4 rename changes only the literal `specflow` → `scaff` in the format
  string at line 304 and the clipboard format at line 513. The escaping call
  (`shell_single_quote_escape(cmd)`) is unchanged; `cmd_escaped` is used
  identically before and after the rename. No new injection surface introduced.

`repo` path:
- Also passed through `shell_single_quote_escape` before interpolation.
- Callers are documented to validate `repo` against the registered-repo set
  before calling `dispatch` (path-traversal guard R9, per doc at line 446).
- Unchanged by T4.

`slug` in comment line:
- `slug` is interpolated unescaped as `{slug}` into a bash **comment** line
  (`# flow-monitor terminal spawn — slug: {slug}`). A slug containing a
  newline could theoretically inject a new shell line after the comment.
- This is a pre-existing condition not introduced by T4; the diff does not
  touch the `{slug}` interpolation or add new validation for it.
- Flagged below as an advisory finding (pre-existing, not introduced by T4,
  but surfaced by this review pass).
